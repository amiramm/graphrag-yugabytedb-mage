# Hybrid Graph RAG on a Single Database: pgvector + MAGE in YugabyteDB 2026.1

*Vector search tells you what's similar. A graph tells you what's connected.
The best retrieval does both — and with YugabyteDB 2026.1 you can do both in
one distributed SQL database, no second system to operate.*

## The problem with vector-only RAG

Retrieval-Augmented Generation usually means one thing: embed your documents,
embed the question, return the nearest chunks by cosine similarity. That works
well when the answer lives in text that *looks like* the question. It works
poorly when the answer depends on **relationships** the embedding can't see:

- *"What else does this product integrate with?"* — the related facts may be
  scattered across chunks that aren't individually similar to the query.
- *"How is A connected to C?"* — when the link runs A → B → C, no single chunk
  is close to a query about A and C together.

Embeddings flatten your knowledge into points in space. A lot of the value is
in the **edges between those points** — and that's exactly what a graph keeps.

**Graph RAG** adds a knowledge graph alongside vector search: entities become
nodes, relationships become edges, and retrieval can *traverse* from a matched
chunk to connected facts. **Hybrid Graph RAG** fuses both signals and hands the
union to the language model.

The usual objection is operational: now you run a vector store *and* a graph
database, keep them in sync, secure two systems, and join across them in
application code. YugabyteDB 2026.1 removes that objection.

## One database, two retrieval modes

YugabyteDB 2026.1 ships two extensions that matter here:

| Extension | What it does |
|-----------|--------------|
| **`vector`** (pgvector-compatible) | `vector` columns + an `ybhnsw` index for fast approximate cosine search |
| **`mage`** | YugabyteDB's Apache AGE-compatible graph engine: property-graph nodes/edges queried with openCypher |

Both run inside the *same* distributed, PostgreSQL-compatible database. Your
chunks and your knowledge graph share one connection, one transaction domain,
one security model, one thing to scale. You cross-reference them with ordinary
SQL.

```
            ┌──────────────────── YugabyteDB 2026.1 (YSQL) ─────────────────────┐
 ingest ──▶ │  doc_chunks(content, embedding vector(1024))  ◀── pgvector ybhnsw  │
            │       │ chunk_id                                                    │
            │       ▼                                                             │
            │  knowledge_graph:  (:Entity {name})-[:RELATED_TO]->(:Entity)        │
            │                      ▲ each node tagged with its source chunk_id    │
            └─────────────────────────────────────────── MAGE ───────────────────┘
 query ──▶  A) vector top-k ─┐
                             ├─▶ fuse ─▶ context for the LLM
            B) graph 1-hop ──┘
```

## Building it

Everything below is from a working demo:
**https://github.com/amiramm/graphrag-yugabytedb-mage**. The code runs against
YugabyteDB 2026.1 and uses Amazon Bedrock (Titan V2 embeddings + Claude) for
embeddings and entity extraction — with a zero-dependency offline fallback so
the pipeline runs without any cloud account.

### 1. The schema — vector table + graph, side by side

```sql
CREATE EXTENSION IF NOT EXISTS vector;
CREATE EXTENSION IF NOT EXISTS mage CASCADE;

CREATE TABLE doc_chunks (
    chunk_id   bigint GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
    doc_id     text         NOT NULL,
    chunk_no   int          NOT NULL,
    content    text         NOT NULL,
    embedding  vector(1024) NOT NULL
);
CREATE INDEX doc_chunks_embedding_idx
    ON doc_chunks USING ybhnsw (embedding vector_cosine_ops);

-- MAGE lives under mag_catalog; create the graph + labels.
SET search_path = mag_catalog, "$user", public;
SELECT create_graph('knowledge_graph');
SELECT create_vlabel('knowledge_graph', 'Entity');
SELECT create_elabel('knowledge_graph', 'RELATED_TO');
```

That's the whole storage layer — a vector table and a property graph in the
same database.

### 2. Ingestion — embed *and* extract

For each chunk we do two things. First, embed it and store the vector:

```python
cur.execute(
    "INSERT INTO doc_chunks (doc_id, chunk_no, content, embedding) "
    "VALUES (%s, %s, %s, %s) RETURNING chunk_id;",
    (doc_id, i, chunk, vec_literal(embed(chunk))),
)
chunk_id = cur.fetchone()[0]
```

Second, extract entity/relationship triples (an LLM call, or a heuristic
offline) and MERGE them into the graph — tagging each node with the
`chunk_id` it came from, so the two stores point at each other:

```python
# one combined MERGE: both endpoints + the edge
cur.execute(
    f"SELECT * FROM cypher('knowledge_graph', $q$ "
    f'MERGE (a:Entity {{name: "{subj}", {tenant}}}) '
    f'MERGE (b:Entity {{name: "{obj}", {tenant}}}) '
    f'MERGE (a)-[r:RELATED_TO {{predicate: "{pred}", {tenant}}}]->(b) '
    "RETURN type(r) $q$) AS (t agtype);"
)
```

### 3. Retrieval — fuse vector and graph

The query path runs both retrievers and combines them. **A. Vector** — nearest
chunks by cosine distance:

```python
cur.execute(
    "SELECT chunk_id, doc_id, content, 1 - (embedding <=> %s) AS cosine_sim "
    "FROM doc_chunks ORDER BY embedding <=> %s LIMIT %s;",
    (q, q, k),
)
```

**B. Graph** — from the entities in those top chunks, traverse one hop in MAGE
to pull in connected facts that vector search alone would miss:

```python
cur.execute(
    f"SELECT * FROM cypher('knowledge_graph', $q$ "
    f'MATCH (a:Entity {{name: "{ent}"}})-[r:RELATED_TO]-(b:Entity) '
    "RETURN r.predicate, b.name $q$) AS (rel agtype, nbr agtype);"
)
```

The fused result — chunks *plus* graph facts — is what you hand to the model.

## It actually runs

```
$ python src/query.py "how does yugabytedb do graph rag?"

--- Vector hits (pgvector) ---
[0.547] yugabytedb.md: YugabyteDB is a distributed SQL database built by Yugabyte...
[0.497] graphrag.md:   Graph RAG combines vector search with a knowledge graph...

--- Graph expansion (MAGE one-hop) ---
  2026.1 Release Line -[ADDS]-> MAGE
  Apache AGE -[COMPATIBLE_WITH]-> MAGE
  Pgvector Extension -[SUPPORTS]-> YugabyteDB
  Knowledge Graph -[FUSES]-> Hybrid Graph RAG
  YSQL API -[RUNS_ON_PORT]-> 5433
  ...
```

Notice what the graph adds. The vector hits surface the two most relevant
paragraphs. The graph then contributes *structured* facts — `Apache AGE
COMPATIBLE_WITH MAGE`, `2026.1 Release Line ADDS MAGE`, `Pgvector Extension
SUPPORTS YugabyteDB` — relationships the LLM can reason over directly, several
of which never appeared in the top chunks.

## Why this matters

This is the same architecture Meko uses in production for agent memory: Mem0
writes facts to pgvector *and* extracts a relationship graph into MAGE, and
memory search fuses both. The pattern generalizes to any RAG system:

- **Operational simplicity.** One database. Vector and graph scale, replicate,
  and fail over together. No sync pipeline, no second security boundary.
- **Richer recall.** Multi-hop relationships and indirect connections that pure
  vector similarity structurally cannot retrieve.
- **It's just SQL + cypher.** No bespoke graph database to learn to operate —
  it's PostgreSQL-compatible YSQL with two extensions.

two databases.*
